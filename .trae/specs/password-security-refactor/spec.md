# 宠物商城密码安全重构 - 产品需求文档

## 概述
- **Summary**: 对宠物商城项目的密码处理进行安全重构，实现密码在前后端传输过程中的加密处理以及数据库中的安全存储。
- **Purpose**: 解决当前项目存在的密码安全隐患，包括前端直接传输明文密码、后端直接存储和比较明文密码的问题。
- **Target Users**: 宠物商城的所有用户，包括普通用户和管理员。

## Goals
- 实现前端密码加密传输，确保密码在网络传输过程中不被窃取
- 实现后端密码加密存储，确保数据库中的密码不被泄露
- 实现安全的密码验证机制，确保用户登录和密码修改的安全性
- 保持系统的兼容性，确保重构后的系统能够正常运行

## Non-Goals (Out of Scope)
- 不改变现有的用户登录流程和界面设计
- 不修改其他功能模块的代码
- 不引入新的依赖库，尽量使用现有的工具类

## Background & Context
- 当前项目前端直接传输明文密码，存在网络传输安全隐患
- 后端直接存储和比较明文密码，存在数据库泄露风险
- 项目使用Spring Boot作为后端框架，Vue作为前端框架
- 后端已有MD5Utils工具类，但未被用于密码加密

## Functional Requirements
- **FR-1**: 前端登录时对密码进行加密处理后再传输
- **FR-2**: 前端注册时对密码进行加密处理后再传输
- **FR-3**: 后端接收加密后的密码并进行验证
- **FR-4**: 后端存储加密后的密码而非明文
- **FR-5**: 后端密码修改时进行安全验证和加密存储

## Non-Functional Requirements
- **NFR-1**: 密码加密算法应使用安全的哈希算法，如MD5或更安全的算法
- **NFR-2**: 加密处理不应显著增加系统响应时间
- **NFR-3**: 加密处理应保持前后端兼容性
- **NFR-4**: 密码验证过程应与原系统保持一致的用户体验

## Constraints
- **Technical**: 使用现有技术栈，不引入新的依赖库
- **Business**: 重构应在不影响现有功能的前提下进行
- **Dependencies**: 依赖后端现有的MD5Utils工具类

## Assumptions
- 后端已有MD5Utils工具类可用于密码加密
- 前端可以使用JavaScript的加密方法或引入简单的加密库
- 重构后系统的性能影响可以忽略不计

## Acceptance Criteria

### AC-1: 前端登录密码加密
- **Given**: 用户在登录页面输入用户名和密码
- **When**: 用户点击登录按钮
- **Then**: 前端对密码进行加密处理后再发送到后端
- **Verification**: `programmatic`
- **Notes**: 加密方式应确保密码在传输过程中不被窃取

### AC-2: 前端注册密码加密
- **Given**: 用户在注册页面输入用户名、密码等信息
- **When**: 用户点击注册按钮
- **Then**: 前端对密码进行加密处理后再发送到后端
- **Verification**: `programmatic`
- **Notes**: 加密方式应与登录时一致

### AC-3: 后端密码加密存储
- **Given**: 后端接收到加密后的密码
- **When**: 后端存储用户信息
- **Then**: 后端将加密后的密码存储到数据库
- **Verification**: `programmatic`
- **Notes**: 数据库中不应存储明文密码

### AC-4: 后端密码验证
- **Given**: 后端接收到登录请求
- **When**: 后端验证用户密码
- **Then**: 后端使用相同的加密算法对输入密码进行加密后与数据库中的密码进行比较
- **Verification**: `programmatic`
- **Notes**: 验证过程应与原系统保持一致的用户体验

### AC-5: 密码修改功能
- **Given**: 用户修改密码
- **When**: 用户提交新密码
- **Then**: 后端对新密码进行加密处理后存储
- **Verification**: `programmatic`
- **Notes**: 密码修改过程应保持安全
